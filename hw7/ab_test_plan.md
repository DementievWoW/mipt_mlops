# План A/B-тестирования ML-модели

## 1. Гипотеза

**H0:** accuracy модели `v1.1.0` не отличается от `v1.0.0`.
**H1:** accuracy `v1.1.0` выше, чем у `v1.0.0`.

## 2. Метрики

- **Первичная (целевая):** accuracy на проде (доля корректных предсказаний от размеченной обратной связи).
- **Вторичная (guardrail):** p95 latency `/predict`, доля 5xx-ошибок.

## 3. Дизайн эксперимента

| Параметр                  | Значение                                                |
|---------------------------|---------------------------------------------------------|
| Юнит рандомизации         | `user_id` (хеш → bucket)                                |
| Доля трафика на v1.1.0    | 50% (контроль 50% / тест 50%)                           |
| Длительность              | 14 дней (учёт недельной сезонности)                     |
| α (уровень значимости)    | 0.05                                                    |
| Power (1 − β)             | 0.80                                                    |
| MDE (минимально детектируемый эффект) | +1.5 п.п. accuracy                          |
| Критерий                  | двусторонний z-тест для пропорций                       |

## 4. Расчёт размера выборки

Для MDE = 1.5 п.п. при базовой accuracy ≈ 95%, α=0.05, power=0.80:

```
n_per_arm ≈ 2 · (z_{1-α/2} + z_{1-β})² · p·(1-p) / MDE²
         ≈ 2 · (1.96 + 0.84)² · 0.95·0.05 / 0.015²
         ≈ 3 300 наблюдений на ветку
```

≈ 6 600 размеченных предсказаний всего.

## 5. Сегментация

- Стратификация по типу клиента (web/mobile) — следим, чтобы не было перекоса.
- Не пускаем в тест health-check трафик и ботов.

## 6. Критерии остановки

- **Успех:** p-value < 0.05 И прирост accuracy ≥ MDE И guardrail не нарушены → раскатываем v1.1.0 на 100%.
- **Откат:** доля 5xx у v1.1.0 > 1% ИЛИ p95 latency вырос > 20% → немедленный откат на blue (см. [ADR-0002](doc/architecture/decisions/0002-use-blue-green-deployment.md)).
- **Нет эффекта:** p-value ≥ 0.05 в конце окна → оставляем v1.0.0.

## 7. Реализация роутинга

Балансировщик (`nginx` или фича-флаг) роутит запрос по хешу `user_id`:

```nginx
split_clients "${remote_addr}${http_x_user_id}" $upstream_pool {
    50%  ml_v110;
    *    ml_v100;
}
```

Логирование: на каждый запрос пишем `(user_id, version, prediction, latency)` → потом матчим с фидбеком и считаем метрики.
